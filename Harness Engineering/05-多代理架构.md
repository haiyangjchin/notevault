# 多代理架构

## 为什么需要多代理？

> "对于更复杂的任务，代理仍然倾向于随着时间的推移而偏离轨道。" - [Anthropic](https://www.anthropic.com/engineering/harness-design-long-running-apps)

**单代理的局限性**：
- 上下文窗口有限
- 无法同时做多件事
- 难以自我评估
- 容易陷入"doom loop"

**多代理的优势**：
- 专业分工
- 并行工作
- 外部验证
- 更好的错误恢复

## 多代理模式

### 1. 初始化代理 + 编码代理

**来源**：[Anthropic](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)

**结构**：
- **初始化代理**：首次运行，设置环境
- **编码代理**：后续运行，增量工作

**职责分离**：
| 代理 | 职责 |
|------|------|
| 初始化代理 | 设置功能列表、进度文件、初始化脚本 |
| 编码代理 | 实现功能、测试、提交、更新进度 |

**优势**：
- 职责清晰
- 专业化
- 易于维护

### 2. 规划器-生成器-评估器（三代理）

**来源**：[Anthropic](https://www.anthropic.com/engineering/harness-design-long-running-apps)

**结构**：
- **规划器**：将简单提示扩展为完整产品规格
- **生成器**：按 sprint 实现功能
- **评估器**：验证工作质量

**职责分离**：
| 代理 | 职责 | 输入 | 输出 |
|------|------|------|------|
| 规划器 | 扩展提示 | 1-4 句话 | 完整产品规格 |
| 生成器 | 实现功能 | 产品规格 | 工作代码 |
| 评估器 | 验证质量 | 工作代码 | 评估报告 |

**工作流程**：
```
用户提示 → 规划器 → 产品规格 → 生成器 → 工作代码 → 评估器 → 评估报告
                                    ↑                              ↓
                                    └──────── 反馈 ────────────────┘
```

### 3. Ralph Wiggum 循环

**来源**：[Geoffrey Huntley](https://ghuntley.com/ralph/)

**结构**：
- 单代理在 while 循环中运行
- 每次循环只做一件事
- 通过提示调优来改进

**代码**：
```bash
while :; do cat PROMPT.md | claude-code ; done
```

**优势**：
- 简单
- 有效
- 易于调试

**劣势**：
- 需要人工监控
- 可能陷入循环
- 需要频繁调优

### 4. 代理团队

**结构**：
- 多个代理并行工作
- 通过共享文件系统协调
- 每个代理有独立的上下文

**协调机制**：
- **共享文件**：代理通过文件交换信息
- **Git**：版本控制和变更跟踪
- **进度文件**：跟踪整体进展

**OpenAI 的实践**：
> "Codex 可以为每次更改启动并驱动一个实例。我们还将 Chrome DevTools 协议接入智能体运行时，并创建了用于处理 DOM 快照、屏幕截图和导航的技能。"

## 代理通信

### 1. 文件通信

**机制**：
- 代理 A 写入文件
- 代理 B 读取文件
- 代理 B 写入响应文件
- 代理 A 读取响应

**示例**：
```
# 代理 A 写入任务
echo "实现用户登录功能" > tasks/current.md

# 代理 B 读取任务并实现
cat tasks/current.md
# ... 实现功能 ...
echo "已完成，测试通过" > tasks/current.md

# 代理 A 读取结果
cat tasks/current.md
```

**优势**：
- 简单
- 可靠
- 易于调试

**劣势**：
- 延迟高
- 需要文件系统访问
- 可能有并发问题

### 2. 消息传递

**机制**：
- 代理通过消息队列通信
- 异步消息传递
- 支持发布/订阅模式

**示例**：
```python
# 代理 A 发送任务
message_queue.publish("tasks", {
    "type": "implement",
    "feature": "user-login",
    "spec": "..."
})

# 代理 B 接收任务
task = message_queue.subscribe("tasks")
# ... 实现功能 ...
message_queue.publish("results", {
    "status": "complete",
    "feature": "user-login"
})
```

**优势**：
- 异步
- 可扩展
- 支持复杂工作流

**劣势**：
- 更复杂
- 需要消息队列基础设施
- 调试更困难

### 3. 共享状态

**机制**：
- 代理共享一个状态存储
- 通过状态读写协调
- 支持事务和锁定

**示例**：
```python
# 代理 A 更新状态
state_store.update("user-login", {
    "status": "in-progress",
    "assigned_to": "agent-a"
})

# 代理 B 读取状态
status = state_store.get("user-login")
if status["status"] == "in-progress":
    # 等待或处理其他任务
    pass
```

**优势**：
- 实时状态
- 支持复杂协调
- 易于监控

**劣势**：
- 需要状态存储
- 可能有并发问题
- 更复杂

## 多代理协调

### 1. 任务分配

**策略**：
- **静态分配**：预先分配任务
- **动态分配**：根据代理能力分配
- **负载均衡**：均匀分配任务

**示例**：
```python
# 静态分配
tasks = [
    {"feature": "user-login", "agent": "agent-a"},
    {"feature": "user-profile", "agent": "agent-b"},
    {"feature": "user-settings", "agent": "agent-c"}
]

# 动态分配
def assign_task(task, agents):
    # 根据代理能力和负载分配
    available_agents = [a for a in agents if a.available]
    best_agent = min(available_agents, key=lambda a: a.current_load)
    return best_agent
```

### 2. 进度跟踪

**机制**：
- **进度文件**：记录整体进展
- **Git 历史**：版本控制
- **状态更新**：实时状态

**示例**：
```markdown
# 进度文件

## 功能列表
- [x] 用户登录
- [x] 用户注册
- [ ] 用户资料
- [ ] 用户设置

## 当前状态
- 代理 A：正在处理用户资料
- 代理 B：正在处理用户设置
- 代理 C：空闲
```

### 3. 错误处理

**策略**：
- **重试**：自动重试失败任务
- **回退**：回退到上一个工作状态
- **升级**：将错误升级给人类

**示例**：
```python
def handle_error(task, error):
    if error.retryable:
        # 自动重试
        task.retry()
    elif error.recoverable:
        # 回退到上一个状态
        task.rollback()
    else:
        # 升级给人类
        escalate_to_human(task, error)
```

## 多代理的最佳实践

### 1. 职责清晰
> "每个代理都有明确的职责，避免职责重叠。" - Anthropic

**实践**：
- 每个代理有明确的输入和输出
- 避免代理之间的依赖循环
- 使用接口定义代理行为

### 2. 通信简单
> "通信通过文件进行：一个代理写入文件，另一个代理读取并响应。" - Anthropic

**实践**：
- 优先使用文件通信
- 避免复杂的通信协议
- 保持通信内容简单

### 3. 状态持久化
> "文件系统和 git 用于跨会话跟踪工作。" - [LangChain](https://www.langchain.com/blog/the-anatomy-of-an-agent-harness)

**实践**：
- 所有状态都持久化到文件系统
- 使用 Git 跟踪变更
- 进度文件记录整体进展

### 4. 错误恢复
> "当事情进行不顺利时，解决方案基本上再也不会是'再努力一点'。" - OpenAI

**实践**：
- 自动重试可恢复的错误
- 回退到上一个工作状态
- 将不可恢复的错误升级给人类

## 实际案例

### Anthropic 的三代理架构

**场景**：构建 2D 复古游戏制作器

**架构**：
- **规划器**：将提示扩展为 16 个功能，跨 10 个 sprint
- **生成器**：按 sprint 实现功能
- **评估器**：使用 Playwright 验证功能

**结果**：
- 6 小时，$200
- 功能完整的 2D 游戏制作器
- 包含精灵编辑器、关卡编辑器、播放模式

### OpenAI 的代理团队

**场景**：构建内部产品

**架构**：
- 3 名工程师
- 多个 Codex 代理并行工作
- 通过 Git 和文件系统协调

**结果**：
- 100 万行代码
- 1500 个 PR
- 平均每位工程师每天 3.5 个 PR

### LangChain 的自验证代理

**场景**：改进 Terminal Bench 排名

**架构**：
- **生成器**：实现功能
- **评估器**：验证功能
- **修复器**：修复问题

**结果**：
- 排名从 Top 30 提升到 Top 5
- 只改变 harness，不改变模型

## 多代理的挑战

### 1. 协调复杂性
- 代理之间的依赖关系
- 死锁和竞态条件
- 状态一致性

### 2. 通信开销
- 文件 I/O 延迟
- 消息传递开销
- 状态同步成本

### 3. 调试困难
- 多个代理同时运行
- 难以追踪问题来源
- 日志分散

### 4. 资源管理
- 代理数量限制
- 内存和 CPU 使用
- 成本控制

## 多代理的未来

### 1. 更智能的协调
- 自动任务分配
- 动态负载均衡
- 智能错误恢复

### 2. 更好的通信
- 更高效的协议

- 更好的并发支持

### 3. 专业化代理
- 测试代理
- 代码审查代理
- 质量保证代理
