# 工具与沙箱

## 为什么需要工具和沙箱？

> "模型（通常）只能处理文本输入并输出文本。它们无法：维护跨交互的持久状态、执行代码、访问实时知识、设置环境和安装包。" - [LangChain](https://www.langchain.com/blog/the-anatomy-of-an-agent-harness)

**核心问题**：
- 模型本身无法与外部世界交互
- 需要工具来执行操作
- 需要沙箱来安全执行

## 工具分类

### 1. 文件系统工具

**目标**：提供持久存储和上下文管理

**工具**：
- `read`：读取文件
- `write`：写入文件
- `edit`：编辑文件
- `delete`：删除文件
- `list`：列出目录

**用途**：
- 读取代码和文档
- 保存工作进度
- 管理配置文件

**Git 工具**：
- `git status`：查看状态
- `git add`：暂存更改
- `git commit`：提交更改
- `git push`：推送更改
- `git pull`：拉取更改
- `git log`：查看历史

**用途**：
- 版本控制
- 协作
- 回滚错误

### 2. 执行工具

**目标**：让代理执行代码和命令

**工具**：
- `bash`：执行 shell 命令
- `python`：执行 Python 代码
- `node`：执行 JavaScript 代码
- `docker`：容器管理

**用途**：
- 运行测试
- 安装依赖
- 构建项目
- 部署应用

**最佳实践**：
- 限制可执行的命令
- 记录所有执行
- 设置超时

### 3. 浏览器工具

**目标**：让代理与 Web 应用交互

**工具**：
- **Playwright MCP**：浏览器自动化
- **Puppeteer MCP**：浏览器自动化
- **Chrome DevTools Protocol**：浏览器调试

**用途**：
- 测试 Web 应用
- 截图验证
- 交互测试
- 性能分析

**示例**：
```javascript
// 使用 Playwright 测试
async function testApp() {
  // 打开浏览器
  await browser.launch();
  
  // 导航到应用
  await page.goto('http://localhost:3000');
  
  // 交互测试
  await page.click('button');
  await page.fill('input', 'test');
  
  // 截图验证
  await page.screenshot({ path: 'test.png' });
  
  // 关闭浏览器
  await browser.close();
}
```

### 4. 监控工具

**目标**：让代理观察和分析应用行为

**工具**：
- **日志**：LogQL、日志文件
- **指标**：PromQL、指标收集器
- **追踪**：TraceQL、分布式追踪

**用途**：
- 运行时行为观察
- 性能监控
- 错误追踪
- 问题诊断

**OpenAI 的实践**：
> "日志、指标和追踪记录会通过一个本地可观测性堆栈展示给 Codex，对任何给定的工作树来说，该堆栈都是临时的。"

### 5. 开发工具

**目标**：提供开发环境支持

**工具**：
- **语言服务器**：代码补全、错误检查
- **包管理器**：依赖管理
- **构建工具**：项目构建
- **测试框架**：自动化测试

**用途**：
- 代码编写
- 依赖管理
- 项目构建
- 测试执行

## 沙箱

### 什么是沙箱？

> "沙箱给代理安全的操作环境。" - [LangChain](https://www.langchain.com/blog/the-anatomy-of-an-agent-harness)

**定义**：沙箱是一个隔离的执行环境，代理可以在其中安全地执行代码和操作。

### 为什么需要沙箱？

**安全考虑**：
- 代理生成的代码可能有恶意行为
- 本地执行风险高
- 需要隔离和限制

**可扩展性**：
- 一个本地环境无法扩展到大型代理工作负载
- 需要按需创建环境
- 工作完成后销毁

### 沙箱的类型

#### 1. 容器沙箱

**技术**：Docker、Podman、containerd

**特点**：
- 轻量级隔离
- 快速启动
- 易于管理

**示例**：
```dockerfile
# Dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
CMD ["npm", "start"]
```

**使用**：
```bash
# 创建沙箱
docker run -d --name agent-sandbox node:18-alpine

# 在沙箱中执行
docker exec agent-sandbox npm test

# 销毁沙箱
docker rm -f agent-sandbox
```

#### 2. 虚拟机沙箱

**技术**：QEMU、VirtualBox、VMware

**特点**：
- 更强的隔离
- 更完整的环境
- 更高的开销

**使用场景**：
- 需要完整操作系统
- 需要特殊硬件
- 需要更强的安全隔离

#### 3. 云沙箱

**技术**：AWS Lambda、Google Cloud Functions、Azure Functions

**特点**：
- 按需创建
- 自动扩展
- 按使用付费

**使用场景**：
- 大规模代理工作负载
- 需要快速扩展
- 成本敏感

#### 4. WebAssembly 沙箱

**技术**：Wasmtime、Wasmer、WasmEdge

**特点**：
- 轻量级
- 跨平台
- 安全隔离

**使用场景**：
- 需要跨平台执行
- 需要轻量级隔离
- 需要高性能

### 沙箱的最佳实践

#### 1. 最小权限原则
- 只授予必要的权限
- 限制文件系统访问
- 限制网络访问

**示例**：
```yaml
# Docker 容器配置
services:
  agent-sandbox:
    image: node:18-alpine
    read_only: true
    tmpfs:
      - /tmp
    cap_drop:
      - ALL
    cap_add:
      - NET_BIND_SERVICE
```

#### 2. 资源限制
- 限制 CPU 使用
- 限制内存使用
- 限制磁盘空间
- 限制执行时间

**示例**：
```bash
# 使用 cgroups 限制资源
docker run -d \
  --cpus="1.0" \
  --memory="512m" \
  --pids-limit=100 \
  --name agent-sandbox \
  node:18-alpine
```

#### 3. 网络隔离
- 限制网络访问
- 只允许必要的网络连接
- 使用代理或网关

**示例**：
```yaml
# Docker 网络配置
services:
  agent-sandbox:
    image: node:18-alpine
    networks:
      - isolated
    dns:
      - 8.8.8.8

networks:
  isolated:
    driver: bridge
    internal: true
```

#### 4. 日志和监控
- 记录所有执行
- 监控资源使用
- 检测异常行为

**示例**：
```bash
# 使用 sysdig 监控容器
sysdig -pc -c spy_users container.name=agent-sandbox
```

## 工具和沙箱的集成

### 1. Claude Agent SDK

**特点**：
- 内置工具支持
- 自动沙箱管理
- 集成开发环境

**工具**：
- 文件系统工具
- 执行工具
- 浏览器工具
- 监控工具

**使用**：
```python
from claude_agent_sdk import Agent

# 创建代理
agent = Agent(
    model="claude-opus-4",
    tools=["filesystem", "bash", "browser"],
    sandbox=True
)

# 执行任务
result = agent.run("构建一个 Web 应用")
```

### 2. LangChain Deep Agents

**特点**：
- 灵活的工具配置
- 可扩展的架构
- 集成多种 LLM

**工具**：
- 文件系统工具
- 执行工具
- 浏览器工具
- 自定义工具

**使用**：
```python
from deepagents import Agent

# 创建代理
agent = Agent(
    model="gpt-5.2-codex",
    tools=[
        FilesystemTool(),
        BashTool(),
        BrowserTool()
    ],
    sandbox=True
)

# 执行任务
result = agent.run("构建一个 Web 应用")
```

### 3. OpenAI Codex CLI

**特点**：
- 命令行界面
- 集成 Git
- 支持多种语言

**工具**：
- 文件系统工具
- 执行工具
- Git 工具

**使用**：
```bash
# 使用 Codex CLI
codex "构建一个 Web 应用"

# 使用沙箱
codex --sandbox "构建一个 Web 应用"
```

## 实际案例

### OpenAI 的 Chrome DevTools 集成

**场景**：让 Codex 测试 Web 应用

**实现**：
1. 将 Chrome DevTools 协议接入代理运行时
2. 创建处理 DOM 快照、屏幕截图和导航的技能
3. 代理可以复现错误、验证修复、推理 UI 行为

**结果**：
- 代理能够自主测试 Web 应用
- 减少了人工测试需求
- 提高了功能完成率

### Anthropic 的 Playwright MCP

**场景**：让 Claude 测试 claude.ai 克隆

**实现**：
1. 使用 Playwright MCP 控制浏览器
2. 像用户一样交互（点击、输入、导航）
3. 截图验证 UI 状态

**结果**：
- 能够识别仅从代码中不明显的 bug
- 显著提高了功能完成率
- 减少了人工测试需求

### LangChain 的沙箱环境

**场景**：在 Terminal Bench 2.0 上运行代理

**实现**：
1. 使用 Harbor 编排运行
2. 旋转沙箱（Daytona）
3. 与代理循环交互
4. 运行验证和评分

**结果**：
- 排名从 Top 30 提升到 Top 5
- 只改变 harness，不改变模型
- 沙箱提供了安全的执行环境

## 工具和沙箱的挑战

### 1. 安全性
- 代理生成的代码可能有恶意行为
- 需要严格的权限控制
- 需要监控和审计

### 2. 性能
- 沙箱有开销
- 工具调用有延迟
- 需要优化资源使用

### 3. 复杂性
- 工具配置复杂
- 沙箱管理复杂
- 需要专业知识

### 4. 成本
- 云沙箱有成本
- 资源使用有成本
- 需要成本控制

## 工具和沙箱的未来

### 1. 更智能的工具
- 自动工具选择
- 动态工具生成
- 工具组合优化

### 2. 更安全的沙箱
- 更强的隔离
- 更细粒度的权限
- 更好的监控

### 3. 更高效的执行
- 更快的启动
- 更低的开销
- 更好的资源利用
