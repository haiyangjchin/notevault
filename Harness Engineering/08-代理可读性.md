# 代理可读性

## 什么是代理可读性？

> "由于该代码仓库完全由智能体生成，我们首先针对 Codex 的可读性进行了优化。" - [OpenAI](https://openai.com/zh-Hans-CN/index/harness-engineering/)

**定义**：代理可读性是指应用程序、日志、指标和追踪对 AI 代理的可见性和可理解性。

**核心思想**：
- 代理看不到的东西就不存在
- 存储在 Google Docs、聊天记录或人们头脑中的知识都无法被系统访问
- 代码仓库本地的、已版本化的工件就是它所能看到的全部

## 为什么需要代理可读性？

### 1. 人工 QA 瓶颈
> "随着代码吞吐量的增加，我们的瓶颈变成了人工 QA 能力。" - OpenAI

**问题**：
- 人类的时间和注意力是固定的限制因素
- 代理生成的输出超过人类能仔细检查的量
- 需要将验证能力扩展到代理

### 2. 代理无法访问的信息
- Google Docs 中的文档
- Slack 消息中的讨论
- 人们头脑中的隐性知识
- 未版本化的配置

### 3. 代理需要直接推理
- 代理需要能够从代码仓库推理出完整的业务领域
- 代理需要能够复现错误、验证修复、推理 UI 行为
- 代理需要能够观察运行时行为

## 代理可读性的层次

### 1. 代码可读性

**目标**：让代理能够理解和修改代码

**实践**：
- 清晰的代码结构
- 有意义的命名
- 充分的注释
- 一致的风格

**示例**：
```python
# 好的代码可读性
def calculate_user_score(user_activity: dict) -> float:
    """
    计算用户活跃度评分
    
    Args:
        user_activity: 用户活动数据，包含登录次数、发布内容数等
        
    Returns:
        float: 用户评分，范围 0-100
    """
    login_score = min(user_activity['logins'] * 10, 50)
    content_score = min(user_activity['posts'] * 5, 30)
    engagement_score = min(user_activity['comments'] * 2, 20)
    
    return login_score + content_score + engagement_score
```

### 2. 文档可读性

**目标**：让代理能够理解项目知识

**实践**：
- 版本化文档
- 结构化格式
- 交叉链接
- 定期更新

**示例**：

### 3. 日志可读性

**目标**：让代理能够理解运行时行为

**实践**：
- 结构化日志
- 有意义的日志级别
- 上下文信息
- 可查询的日志

**示例**：
```json
{
  "timestamp": "2026-06-02T10:30:00Z",
  "level": "INFO",
  "service": "user-service",
  "method": "POST",
  "path": "/api/users",
  "user_id": "12345",
  "action": "create_user",
  "result": "success",
  "duration_ms": 150,
  "trace_id": "abc-123-def"
}
```

**工具**：
- **LogQL**：查询日志
- **PromQL**：查询指标
- **TraceQL**：查询追踪

### 4. 指标可读性

**目标**：让代理能够监控系统健康

**实践**：
- 关键指标监控
- 告警规则
- 仪表板
- 历史趋势

**示例**：
```yaml
# 指标定义
metrics:
  - name: http_requests_total
    help: Total HTTP requests
    labels: [method, path, status]
    
  - name: http_request_duration_seconds
    help: HTTP request duration
    labels: [method, path]
    
  - name: active_users
    help: Number of active users
```

**查询**：
```promql
# 查询请求率
rate(http_requests_total[5m])

# 查询错误率
rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m])

# 查询响应时间
histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))
```

### 5. 追踪可读性

**目标**：让代理能够理解请求流程

**实践**：
- 分布式追踪
- 跨服务追踪
- 性能分析
- 错误追踪

**示例**：
```json
{
  "trace_id": "abc-123-def",
  "span_id": "span-1",
  "parent_span_id": null,
  "service": "api-gateway",
  "operation": "POST /api/orders",
  "start_time": "2026-06-02T10:30:00Z",
  "end_time": "2026-06-02T10:30:01Z",
  "duration_ms": 1000,
  "status": "OK",
  "tags": {
    "user_id": "12345",
    "order_id": "order-789"
  }
}
```

**工具**：
- **OpenTelemetry**：追踪收集
- **Jaeger**：追踪存储和查询
- **Zipkin**：追踪可视化

## 代理可读性的实践

### 1. Chrome DevTools 集成

**OpenAI 的实践**：
> "我们还将 Chrome DevTools 协议接入智能体运行时，并创建了用于处理 DOM 快照、屏幕截图和导航的技能。"

**实现**：
1. 将 Chrome DevTools 协议接入代理运行时
2. 创建处理 DOM 快照、屏幕截图和导航的技能
3. 代理可以复现错误、验证修复、推理 UI 行为

**用途**：
- 测试 Web 应用
- 截图验证
- 交互测试
- 性能分析

**示例**：
```javascript
// 使用 Chrome DevTools 测试
async function testWithDevTools() {
  // 连接到浏览器
  const client = await CDP({ port: 9222 });
  
  // 导航到页面
  await client.Page.navigate({ url: 'http://localhost:3000' });
  
  // 等待页面加载
  await client.Page.loadEventFired();
  
  // 截图
  const screenshot = await client.Page.captureScreenshot();

  const dom = await client.DOM.getDocument();

    expression: 'document.title'
  });
}
```

### 2. 可观测性堆栈

**OpenAI 的实践**：
> "日志、指标和追踪记录会通过一个本地可观测性堆栈展示给 Codex，对任何给定的工作树来说，该堆栈都是临时的。"

**架构**：
```
应用程序 → Vector → 可观测性堆栈
├── Victoria Logs (日志)
├── Victoria Metrics (指标)
└── Victoria Traces (追踪)
```

**查询**：
- **LogQL**：查询日志
- **PromQL**：查询指标
- **TraceQL**：查询追踪

**用途**：
- 代理可以查询日志
- 代理可以查询指标
- 代理可以查询追踪
- 代理可以关联和推理

### 3. 临时环境

**OpenAI 的实践**：
> "Codex 在该应用程序的一个完全独立的版本上运行，一旦任务完成，该版本的所有内容，包括日志和指标，都会被删除。"

**实现**：
1. 为每个任务创建独立的环境
2. 包含应用程序、日志、指标、追踪
3. 任务完成后销毁环境

**优势**：
- 隔离性：不同任务互不干扰
- 清洁性：任务完成后环境被销毁
- 可重复性：可以重复创建相同环境

## 代理可读性的最佳实践

### 1. 结构化数据
- 使用 JSON 格式
- 包含上下文信息
- 使用有意义的字段名

**示例**：
```json
{
  "timestamp": "2026-06-02T10:30:00Z",
  "level": "INFO",
  "service": "user-service",
  "action": "create_user",
  "user_id": "12345",
  "result": "success",
  "duration_ms": 150
}
```

### 2. 可查询的日志
- 使用查询语言
- 支持过滤和聚合
- 支持时间范围查询

**示例**：
```logql
{service="user-service"} |= "create_user" | logfmt | duration_ms > 100
```

### 3. 可视化仪表板
- 关键指标可视化
- 实时更新
- 告警规则

**示例**：
```yaml
# Grafana 仪表板
dashboard:
  title: User Service
  panels:
    - title: Request Rate
      type: graph
      query: rate(http_requests_total{service="user-service"}[5m])
      
    - title: Error Rate
      type: graph
      query: rate(http_requests_total{service="user-service", status=~"5.."}[5m])
      
    - title: Response Time
      type: graph
      query: histogram_quantile(0.95, rate(http_request_duration_seconds_bucket{service="user-service"}[5m]))
```

### 4. 追踪上下文
- 跨服务追踪
- 包含业务上下文
- 支持错误追踪

**示例**：
```json
{
  "trace_id": "abc-123-def",
  "span_id": "span-1",
  "service": "user-service",
  "operation": "create_user",
  "tags": {
    "user_id": "12345",
    "email": "user@example.com"
  },
  "logs": [
    {
      "timestamp": "2026-06-02T10:30:00Z",
      "message": "Creating user",
      "level": "INFO"
    },
    {
      "timestamp": "2026-06-02T10:30:01Z",
      "message": "User created successfully",
      "level": "INFO"
    }
  ]
}
```

## 实际案例

### OpenAI 的代理可读性实践

**场景**：构建内部产品

**实践**：
1. Chrome DevTools 集成：让 Codex 测试 Web 应用
2. 可观测性堆栈：让 Codex 查询日志、指标、追踪
3. 临时环境：为每个任务创建独立环境

**结果**：
- 代理能够自主测试 Web 应用
- 代理能够查询和分析运行时数据

### Anthropic 的 Playwright 实践

**场景**：测试 claude.ai 克隆

**实践**：
1. 使用 Playwright MCP 控制浏览器
2. 像用户一样交互
3. 截图验证 UI 状态

**结果**：
- 能够识别仅从代码中不明显的 bug
- 显著提高了功能完成率
- 减少了人工测试需求

## 代理可读性的挑战

### 1. 数据量
- 日志、指标、追踪数据量大
- 需要高效的存储和查询
- 需要数据压缩和归档

### 2. 实时性
- 代理需要实时数据
- 需要低延迟的查询
- 需要实时的可视化

### 3. 复杂性
- 多种数据源
- 多种查询语言
- 多种可视化工具

### 4. 成本
- 存储成本
- 计算成本
- 网络成本

- 统一的数据模型
- 跨工具集成
- 自动化工作流
