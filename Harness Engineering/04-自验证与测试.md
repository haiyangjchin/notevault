# 自验证与测试

## 为什么自验证很重要？

> "今天的模型是出色的自我改进机器。自验证允许代理在运行中通过反馈进行自我改进。" - [LangChain](https://www.langchain.com/blog/improving-deep-agents-with-harness-engineering)

**核心问题**：
- 代理倾向于对自己的工作过度自信
- 代理可能标记功能为完成，而没有适当测试
- 代理可能忽略边界情况

**解决方案**：构建自验证循环，让代理能够自我评估和改进。

## 自验证循环

### 基本模式
```
1. 构建功能
2. 运行测试
3. 检查结果
4. 如果失败，修复问题
5. 重复直到通过
```

### LangChain 的实践

LangChain 在系统提示中添加了如何解决问题的指导：

#### 1. 规划和发现
- 阅读任务
- 扫描代码库
- 基于任务规格和验证方式构建初始计划

#### 2. 构建
- 实现计划，同时考虑验证
- 构建测试（如果不存在）
- 测试正常路径和边界情况

#### 3. 验证
- 运行测试
- 阅读完整输出
- 与要求对比（而非自己的代码）

#### 4. 修复
- 分析任何错误
- 重新审视原始规格
- 修复问题

## 验证工具

### 1. 浏览器自动化

**工具**：Playwright MCP、Puppeteer MCP、Chrome DevTools Protocol

**用途**：
- 像真实用户一样测试 Web 应用
- 截图验证 UI 状态
- 交互测试（点击、输入、导航）

**Anthropic 的实践**：
> "提供 Claude 这些测试工具显著提高了性能，因为代理能够识别和修复仅从代码中不明显的 bug。"

**示例**：
```javascript
// 使用 Playwright 测试
async function testNewChat() {
  // 导航到主界面
  await page.goto('http://localhost:3000');
  
  // 点击新建聊天按钮
  await page.click('[data-testid="new-chat"]');
  
  // 验证新对话已创建
  const chatArea = await page.$('[data-testid="chat-area"]');
  expect(chatArea).toHaveText('Welcome');
  
  // 验证对话出现在侧边栏
  const sidebar = await page.$('[data-testid="sidebar"]');
  expect(sidebar).toContainText('New Chat');
}
```

### 2. 测试运行器

**工具**：单元测试框架、集成测试框架

**用途**：
- 自动化测试执行
- 覆盖率报告
- 回归测试

**最佳实践**：
- 测试应该快速（毫秒到秒级）
- 测试应该确定性（相同输入，相同输出）
- 测试应该独立（不依赖外部状态）

### 3. 静态分析

**工具**：Linter、类型检查器、代码分析器

**用途**：
- 代码风格检查
- 类型安全验证
- 架构约束检查

**示例**：
- ESLint：JavaScript/TypeScript 代码检查
- TypeScript：类型检查
- ArchUnit：架构约束测试

### 4. 日志和监控

**工具**：LogQL、PromQL、OpenTelemetry

**用途**：
- 运行时行为观察
- 性能监控
- 错误追踪

**OpenAI 的实践**：
> "日志、指标和追踪记录会通过一个本地可观测性堆栈展示给 Codex，对任何给定的工作树来说，该堆栈都是临时的。"

## 计算型 vs 推理型验证

根据 [Birgitta Böckeler 的分析](https://martinfowler.com/articles/harness-engineering.html)，验证分为两类：

### 计算型验证（Computational）
- **特点**：确定性、快速、可靠
- **示例**：测试、linter、类型检查、结构分析
- **时间**：毫秒到秒
- **成本**：低

**用途**：
- 代码风格检查
- 类型安全验证
- 架构约束检查
- 单元测试

### 推理型验证（Inferential）
- **特点**：语义分析、更慢、更贵、非确定性
- **示例**：AI 代码审查、LLM 作为评判者
- **时间**：秒到分钟
- **成本**：高

**用途**：
- 语义重复代码检测
- 冗余测试识别
- 过度工程检测
- 设计质量评估

**权衡**：
- 计算型便宜、快速、确定性，但只能捕获结构问题
- 推理型更贵、更慢、非确定性，但能捕获语义问题
- 优先使用计算型，必要时才用推理型

## 验证的最佳实践

### 1. 测试驱动开发
> "我们添加了关于如何解决问题的指导到系统提示中。" - LangChain

**流程**：
1. 先写测试
2. 实现功能
3. 运行测试
4. 修复失败

### 2. 真实环境测试
> "Claude 主要在验证功能端到端时做得很好，一旦被明确提示使用浏览器自动化工具并像人类用户一样进行所有测试。" - Anthropic

**实践**：
- 使用浏览器自动化测试 Web 应用
- 像真实用户一样交互
- 验证完整的工作流程

### 3. 边界情况测试
> "提示强调边界情况测试，帮助代理避免只检查'正常路径'情况。" - LangChain

**实践**：
- 测试正常路径
- 测试边界情况
- 测试错误情况
- 测试性能边界

### 4. 预完成检查清单

**LangChain 的实践**：
```python
class PreCompletionChecklistMiddleware:
    def on_agent_exit(self):
        # 拦截代理退出
        # 提醒它运行验证
        # 对照任务规格检查
        pass
```

**目的**：
- 防止代理过早退出
- 确保所有功能都经过验证
- 对照原始规格检查

## 验证的挑战

### 1. 代理的过度自信
> "当被要求评估自己的工作时，代理倾向于自信地赞扬工作——即使对人类观察者来说质量明显平庸。" - [Anthropic](https://www.anthropic.com/engineering/harness-design-long-running-apps)

**解决方案**：
- 分离生成器和评估器
- 训练评估器保持怀疑态度
- 使用外部反馈循环

### 2. 视觉限制
> "Claude 的视觉和浏览器自动化工具的限制使得难以识别每种 bug。" - Anthropic

**示例**：
- Claude 看不到浏览器原生的 alert 模态框
- 依赖这些模态框的功能往往更 buggy

### 3. 测试质量
> "这种方法对 AI 生成的测试给予了很大的信任，这还不够好。" - [Birgitta Böckeler](https://martinfowler.com/articles/harness-engineering.html)

**挑战**：
- AI 生成的测试可能不够全面
- 测试可能遗漏边界情况
- 测试可能不够深入

**解决方案**：
- 使用 approved fixtures 模式
- 结合人工审查
- 使用变异测试评估测试质量

## 实际案例

### Anthropic 的浏览器自动化测试

**场景**：测试 claude.ai 克隆

**工具**：Puppeteer MCP

**流程**：
1. 代理启动开发服务器
2. 使用 Puppeteer 打开浏览器
3. 像用户一样交互（点击、输入、导航）
4. 截图验证 UI 状态
5. 识别和修复 bug

**结果**：
- 能够识别仅从代码中不明显的 bug
- 显著提高了功能完成率
- 减少了人工测试需求

### LangChain 的自验证改进

**场景**：改进 Terminal Bench 2.0 排名

**方法**：
1. 添加自验证指导到系统提示
2. 使用 `PreCompletionChecklistMiddleware`
3. 强调测试重要性

**结果**：
- 排名从 Top 30 提升到 Top 5
- 只改变 harness，不改变模型
- 自验证是关键改进

5. **持续改进**：每次错误都是改进机会
