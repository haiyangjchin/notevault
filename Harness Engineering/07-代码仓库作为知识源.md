# 代码仓库作为知识源

## 核心原则

> "代码仓库本地的、已版本化的工件（例如，代码、Markdown、模式、执行计划）就是它所能看到的全部。" - [OpenAI](https://openai.com/zh-Hans-CN/index/harness-engineering/)

**核心思想**：
- 代理看不到的东西就不存在
- 存储在 Google Docs、聊天记录或人们头脑中的知识都无法被系统访问
- 代码仓库就是代理的"世界"

### 文档类型

#### 1. 设计文档（Design Docs）
- **目的**：记录架构决策和设计理由
- **内容**：问题描述、解决方案、权衡分析
- **位置**：`docs/design-docs/`

**示例**：
```markdown
# 用户认证设计文档

## 问题
需要实现用户认证系统，支持多种认证方式。

## 解决方案
使用 JWT + OAuth2.0 组合方案。

## 权衡
- 优点：无状态、可扩展
- 缺点：需要管理密钥

## 决策
采用 JWT + OAuth2.0 方案。
```

#### 2. 执行计划（Exec Plans）
- **目的**：记录复杂工作的计划和进度
- **内容**：任务列表、进度、决策日志
- **位置**：`docs/exec-plans/`

**示例**：
```markdown
# 用户认证执行计划

## 任务列表
- [x] 设计认证架构
- [ ] 实现 JWT 服务
- [ ] 实现 OAuth2.0 集成
- [ ] 编写测试
- [ ] 部署到生产环境

## 进度
- 2026-06-01：完成设计文档
- 2026-06-02：开始实现 JWT 服务

## 决策日志
- 2026-06-01：选择 JWT + OAuth2.0 方案
```

#### 3. 产品规格（Product Specs）
- **目的**：记录产品需求和功能规格
- **内容**：用户故事、功能描述、验收标准
- **位置**：`docs/product-specs/`

**示例**：
```markdown
# 用户登录功能

## 用户故事
作为用户，我想能够登录系统，以便访问我的个人资料。

## 功能描述
- 支持邮箱/密码登录
- 支持 OAuth 登录（Google、GitHub）
- 支持两步验证

## 验收标准
- [ ] 用户可以使用邮箱和密码登录
- [ ] 用户可以使用 Google 账号登录
- [ ] 用户可以使用 GitHub 账号登录
- [ ] 用户可以启用两步验证
```

#### 4. 参考资料（References）
- **目的**：记录外部库和工具的使用指南
- **内容**：API 文档、配置示例、最佳实践
- **位置**：`docs/references/`

**示例**：
```markdown
# Stripe API 参考

## 基础配置
```javascript
const stripe = require('stripe')('sk_test_...');
```

## 创建支付意图
```javascript
const paymentIntent = await stripe.paymentIntents.create({
  amount: 1000,
  currency: 'usd',
});
```

## 最佳实践
- 使用 Webhook 处理异步事件
- 实现幂等性
- 处理错误和重试
```

#### 5. 质量文档
- **目的**：记录质量标准和评分
- **内容**：质量指标、评分标准、改进计划
- **位置**：`docs/QUALITY_SCORE.md`

**示例**：
```markdown
# 质量评分

## 评分标准
- 代码风格：10 分
- 测试覆盖率：20 分
- 文档完整性：15 分
- 性能指标：25 分
- 用户体验：30 分

## 当前评分
- 总分：85/100
- 代码风格：9/10
- 测试覆盖率：18/20
- 文档完整性：12/15
- 性能指标：22/25
- 用户体验：24/30

## 改进计划
- 提高测试覆盖率到 95%
- 完善文档
- 优化性能
```

## AGENTS.md 的设计

### 作为"地图"而非"百科全书"

> "我们尝试了'一个大型 AGENTS.md'方法。可想而知，这是一次失败的尝试。" - OpenAI

**问题**：
- 情境是一种稀缺资源
- 一个巨大的指令文件会挤掉任务、代码和相关文档
- 智能体要么会错过关键约束条件，要么开始针对错误的约束条件进行优化

**解决方案**：将 AGENTS.md 作为"内容目录"

### AGENTS.md 的结构

```markdown
# AGENTS.md

## 项目概述
这是一个 Web 应用项目，使用 React + Node.js + PostgreSQL。

## 代码仓库结构
- `src/`：源代码
- `docs/`：文档
- `tests/`：测试
- `config/`：配置

## 关键文档
- [架构文档](docs/ARCHITECTURE.md)
- [设计文档](docs/design-docs/index.md)
- [产品规格](docs/product-specs/index.md)
- [质量评分](docs/QUALITY_SCORE.md)

## 开发指南
- 使用 TypeScript
- 遵循 ESLint 规则
- 编写单元测试
- 提交前运行测试

## 工具
- Git：版本控制
- npm：包管理
- Jest：测试框架
- ESLint：代码检查
```

### AGENTS.md 的最佳实践

#### 1. 保持简短
- 约 100 行
- 只包含最重要的信息
- 深层信息指向其他文档

#### 2. 作为索引
- 指向详细文档
- 提供导航
- 不要重复信息

#### 3. 易于维护
- 定期更新
- 删除过时信息
- 保持一致性

## 知识管理

### 1. 版本控制

**目的**：跟踪知识变更

**实践**：
- 所有文档都在 Git 中
- 使用提交消息记录变更
- 使用分支管理不同版本

**示例**：
```bash
# 添加文档
git add docs/ARCHITECTURE.md
git commit -m "docs: 添加架构文档"

# 更新文档
git add docs/QUALITY_SCORE.md
git commit -m "docs: 更新质量评分"
```

### 2. 交叉链接

**目的**：建立文档之间的关系

**实践**：
- 使用相对链接
- 建立索引
- 定期检查链接有效性

**示例**：
```markdown
# 架构文档

## 相关文档
- [设计文档](design-docs/index.md)
- [产品规格](product-specs/index.md)
- [质量评分](QUALITY_SCORE.md)
```

### 3. 新鲜度检查

**目的**：确保文档反映当前状态

**实践**：
- 定期扫描过时文档
- 自动化检查
- 发起修复 PR

**OpenAI 的实践**：
> "一个定期运行的'doc-gardening'智能体会扫描那些不再反映真实代码行为的过时或废弃文档，并发起修复用的 Pull Request。"

### 4. 质量评分

**目的**：衡量文档质量

**实践**：
- 定义评分标准
- 定期评估
- 跟踪改进

**示例**：
```markdown
# 文档质量评分

## 评分标准
- 完整性：是否覆盖所有主题？
- 准确性：信息是否正确？
- 新鲜度：是否反映当前状态？
- 可读性：是否易于理解？

## 当前评分
- 架构文档：95/100
- 设计文档：85/100
- 产品规格：90/100
- 质量评分：80/100

## 改进计划
- 完善设计文档
- 更新质量评分
```

## 代码仓库的治理

### 1. 自动化检查

**工具**：
- **Linter**：检查代码风格
- **类型检查器**：检查类型安全
- **架构检查器**：检查架构约束
- **文档检查器**：检查文档质量

**实践**：
- 在 CI 中运行检查
- 自动化修复
- 强制执行规则

**OpenAI 的实践**：
> "专职的 linter 和 CI 作业会验证知识库的更新状况、是否已交叉链接且结构正确。"

### 2. 代码审查

**目的**：确保代码质量

**实践**：
- 自动化审查
- 人工审查
- 持续改进

**OpenAI 的实践**：
> "人类可以审核 Pull Request（合并请求），但并非必须这样做。随着时间的推移，我们已将几乎所有的审核工作调整为用智能体对智能体的方式来处理。"

### 3. 技术债务管理

**目的**：跟踪和管理技术债务

**实践**：
- 记录技术债务
- 优先级排序
- 定期偿还

**OpenAI 的实践**：
> "技术债务就像一笔高息贷款：不断地以小额贷款的方式偿还债务，总比让债务不断累积，再痛苦地一次解决要好得多。"

## 实际案例

### OpenAI 的代码仓库管理

**场景**：构建内部产品

**实践**：
1. 代码仓库作为"记录系统"
2. AGENTS.md 作为"地图"
3. docs/ 目录作为知识库
4. 自动化检查和治理

**结果**：
- 100 万行代码
- 1500 个 PR
- 平均每位工程师每天 3.5 个 PR

### Anthropic 的进度文件

**场景**：长时间运行代理

**实践**：
1. 进度文件记录工作历史
2. Git 历史跟踪变更
3. 功能列表管理任务

**结果**：
- 代理能够跨会话工作
- 减少了重复工作
- 提高了任务完成率

## 代码仓库的挑战

### 1. 信息过载
- 文档太多
- 难以找到相关信息
- 需要更好的组织

### 2. 信息过时
- 文档不反映当前状态
- 需要定期更新
- 需要自动化检查

### 3. 信息质量
- 文档质量参差不齐
- 需要质量标准
- 需要持续改进

### 4. 信息访问
- 代理需要快速访问
- 需要索引和搜索
- 需要缓存和优化
