# 上下文工程

## 什么是上下文工程？

上下文工程是 **关于在特定时刻让任务对模型来说是可解决的**。

> "上下文工程是关于在给定时刻让任务对模型来说是可解决的。你给它正确的文本，让它能够自信地回答。" - [Louis-François Bouchard](https://www.youtube.com/watch?v=zYerCzIexCg)

## 上下文工程 vs 提示工程

| 方面 | 提示工程 | 上下文工程 |
|------|---------|-----------|
| **范围** | 单次交互 | 整个上下文窗口 |
| **焦点** | 问什么 | 发送什么给模型 |
| **目标** | 获得好的回答 | 让任务可解决 |
| **技术** | 提示模板、示例 | 信息检索、压缩、管理 |

## 上下文窗口的挑战

### 1. 上下文腐烂（Context Rot）
> "上下文腐烂描述了模型在上下文窗口填满时推理和完成任务的能力下降。" - [LangChain](https://www.langchain.com/blog/the-anatomy-of-an-agent-harness)

**症状**：
- 推理质量下降
- 忽略重要信息
- 过早结束工作（"上下文焦虑"）

### 2. 信息过载
- 噪音淹没信号
- 重要信息被忽略
- 模式匹配而非有意识导航

### 3. 状态丢失
- 跨会话信息丢失
- 重复工作
- 决策不一致

## 上下文工程策略

### 1. 渐进式披露（Progressive Disclosure）

**原则**：不要一开始就加载所有信息

**实现**：
- **AGENTS.md 作为"地图"**：简短的索引（约 100 行），指向更深层的信息
- **深层信息按需加载**：只在需要时才读取详细文档
- **技能系统**：技能前置信息按需加载

**OpenAI 的实践**：
> "我们尝试了'一个大型 AGENTS.md'方法。可想而知，这是一次失败的尝试：情境是一种稀缺资源。一个巨大的指令文件会挤掉任务、代码和相关文档。"

**更好的方法**：
```
AGENTS.md (地图，约 100 行)
├── 指向 docs/ARCHITECTURE.md
├── 指向 docs/DESIGN.md
├── 指向 docs/QUALITY_SCORE.md
└── 指向其他深层文档
```

### 2. 压缩（Compaction）

**目标**：在上下文窗口接近满时智能卸载和总结

**技术**：
- **工具调用卸载**：保留工具输出的头尾 tokens，完整输出卸载到文件系统
- **对话总结**：总结早期对话内容
- **选择性保留**：只保留最重要的信息

### 3. 环境上下文注入

**目标**：在代理开始工作前注入环境信息

**LangChain 的实践**：

#### 目录上下文和工具发现
```python
class LocalContextMiddleware:
    def on_agent_start(self):
        # 映射当前目录
        context = map_directory_structure(self.cwd)
        # 发现可用工具
        tools = discover_available_tools()
        # 注入到上下文
        self.inject_context(context, tools)
```

#### 时间预算管理
- 注入时间预算警告
- 提醒代理完成工作并转向验证
- 帮助代理在时间限制内工作

#### 测试要求说明
- 说明工作将通过程序化测试验证
- 强调边界情况测试
- 要求遵循文件路径规范

### 4. 上下文重置（Context Reset）

**目标**：在上下文窗口填满时完全清除并重新开始

**与压缩的区别**：
| 方面 | 压缩 | 重置 |
|------|------|------|
| **连续性** | 保持 | 中断 |
| **上下文焦虑** | 可能持续 | 消除 |
| **状态传递** | 就地总结 | 结构化交接 |
| **复杂度** | 较低 | 较高（需要交接） |

**何时使用**：
- 模型表现出"上下文焦虑"
- 压缩后质量仍然下降
- 需要"干净的开始"

### 5. 文件系统作为上下文扩展

**原则**：文件系统是自然的上下文扩展

**实践**：
- **进度文件**：记录工作历史
- **功能列表**：JSON 格式的待办事项
- **Git 历史**：版本控制和变更跟踪
- **文档**：项目知识库

## 上下文工程的最佳实践

### 1. 地图而非百科全书
> "要给 Codex 的是一张地图，而不是一本 1,000 页的说明书。" - OpenAI

**实现**：
- AGENTS.md 保持简短（约 100 行）
- 主要作为索引和导航
- 深层信息在 docs/ 目录中

### 2. 代码仓库作为记录系统
> "代码仓库的知识库位于一个结构化了的 docs/ 目录中，此目录被当作记录系统来使用。" - OpenAI

**结构**：
```
docs/
├── design-docs/          # 设计文档
├── exec-plans/           # 执行计划
├── generated/            # 生成的文档
├── product-specs/        # 产品规格
├── references/           # 参考资料
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

### 3. 强制执行知识质量
- **专职 linter**：验证知识库的更新状况
- **CI 作业**：检查交叉链接和结构
- **doc-gardening 代理**：扫描过时文档并发起修复 PR

### 4. 代理可读性优化
> "由于该代码仓库完全由智能体生成，我们首先针对 Codex 的可读性进行了优化。" - OpenAI

**原则**：
- 代理看不到的东西就不存在
- 存储在 Google Docs、聊天记录或人们头脑中的知识都无法被系统访问
- 代码仓库本地的、已版本化的工件就是它所能看到的全部

## 上下文工程的权衡

### 信息量 vs 质量
- **太多信息**：噪音淹没信号，模型模式匹配而非有意识导航
- **太少信息**：模型缺乏必要知识，无法完成任务

### 静态 vs 动态
- **静态配置**：简单但可能过时
- **动态发现**：灵活但更复杂

### 全局 vs 局部
- **全局上下文**：项目级别，可能不相关
- **局部上下文**：任务级别，更精确

## 实际案例

### OpenAI 的渐进式披露

**问题**：大型 AGENTS.md 导致上下文过载

**解决方案**：
1. 将 AGENTS.md 缩短到约 100 行
2. 作为"地图"指向深层文档
3. docs/ 目录作为"记录系统"
4. 强制执行知识质量

**结果**：
- 代理能够更好地导航
- 减少了上下文过载
- 提高了任务完成率

### LangChain 的环境上下文注入

**问题**：代理不了解环境，导致错误

**解决方案**：
1. `LocalContextMiddleware`：映射目录和工具
2. 时间预算警告：提醒代理时间限制
3. 测试要求说明：强调验证重要性

**结果**：
- 减少了环境相关错误
- 提高了测试覆盖率
- 更好的时间管理

## 上下文工程的工具

### 信息检索
- **向量数据库**：语义搜索相关文档
- **关键词搜索**：精确匹配
- **知识图谱**：关系和连接

### 信息压缩
- **总结模型**：自动总结长文本
- **提取模型**：提取关键信息
- **过滤器**：移除噪音

### 信息管理
- **版本控制**：跟踪信息变更
- **访问控制**：管理信息可见性
- **新鲜度检查**：确保信息最新
